# Testing Candidates â€” Changes Since March 2026

This document catalogs code changes introduced in commits since **March 2026** that would
benefit from automated testing. It was produced by auditing the 30 commits on this fork
(Spoolman NG) and identifying every meaningful unit of logic â€” decoders, parsers, filters,
aggregations, hooks, endpoints, and bug fixes â€” that currently lacks test coverage or whose
regression risk warrants a dedicated test.

## How to read this

- **Commit** â€” short hash + PR number.
- **Area / File** â€” file path plus the function/class/component when identifiable.
- **Change** â€” what was added or changed.
- **Testing gap / Risk** â€” why it needs a test / what could break.
- **Suggested tests** â€” concrete test ideas.
- **Type** â€” `unit` / `integration` / `component` / `e2e`.
- **Priority** â€” `High` / `Med` / `Low`.

## Test infrastructure (current state)

- **Backend (Python):** `pytest`. Unit tests live in `tests/` (only `test_math.py`,
  `test_extra_field_validation.py` today). Integration tests (real DB + running server, run
  across SQLite/Postgres/MariaDB/CockroachDB) live in `tests_integration/tests/`.
- **Frontend (React/TypeScript, `client/`):** **no test runner and zero test files.** Every
  frontend row below assumes **Vitest + jsdom** (and React Testing Library for `component`
  rows) would first need to be set up. Several inline helpers are not exported and would need
  a small extraction refactor to be unit-testable.
- The NFC modules, `spoolman/client.py`, `spoolman/externaldb.py`, and `spoolman/bump.py`
  have **no tests at all**.
